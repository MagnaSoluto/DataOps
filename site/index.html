DevOps & DataOps, na prática.

Uma síntese do que realmente importa para entregar com qualidade: cultura, automação, ciclos (SDLC), testes antecipados (Shift Left), pipelines CI/CD e runbooks para responder a incidentes com consistência.

Mapa mental em 4 frases

DevOps derruba o muro entre desenvolvimento e operações. DataOps derruba o muro entre engenharia de dados e quem consome dados. CI/CD automatiza validações e deploy. Runbooks reduzem improviso em incidentes.

Checklist de uma entrega “DevOps”

  • 1 Versionar com Git e trabalhar em pequenas mudanças.
  • 2 Automatizar validações (CI): checagens, testes e padrões.
  • 3 Publicar (CD): deploy repetível e rastreável.
  • 4 Operar: logs/monitoramento + runbook para incidentes.

Da cascata à engenharia moderna

O ponto central da evolução foi reduzir custo de mudança e detectar erros cedo. O DevOps não é “uma ferramenta”, é uma forma de trabalhar com responsabilidade compartilhada.

DevOps DataOps Shift Left

Por que o Waterfall falha em software comercial?

Porque requisitos mudam. Em cascata, testes e validações costumam acontecer tarde, então o custo de corrigir aumenta e a equipe sofre com “surpresas” perto do prazo.

rigidez testes tardios custo 50× no final

DevOps: unir o que nunca deveria ter sido separado

DevOps integra Dev e Ops para reduzir tempo entre desenvolvimento e entrega, aumentar confiabilidade e automatizar o ciclo de vida: build, testes, deploy e monitoramento.

cultura automação feedback contínuo

Onde entra o DataOps?

Se no DevOps o problema era o muro entre Dev e Ops, no DataOps o problema é o muro entre engenharia de dados e consumidores de dados (cientistas, analistas e negócio). DataOps aplica os mesmos princípios ao fluxo de valor de dados: automação de pipelines, qualidade, governança e colaboração.

CALMS: os pilares do DevOps

Sem CALMS, DevOps vira só ferramenta. Com CALMS, vira transformação sustentável.

C — Culture

Confiança, colaboração, comunicação e responsabilidade compartilhada.

A — Automation

Automatizar pipeline CI/CD, testes, deploy e tarefas repetitivas com rastreabilidade.

L — Lean

Trabalhar em pequenos lotes, reduzir desperdício e manter foco no fluxo de valor.

M — Measurement

Medir para melhorar: métricas, feedback loops e decisões guiadas por dados.

S — Sharing

Compartilhar conhecimento, ferramentas, métricas e aprendizados entre equipes.

Resultado esperado

Entregas mais rápidas e confiáveis, com menos retrabalho e menos incidentes em produção.

SDLC + ciclo DevOps: o “mapa” da entrega

O SDLC organiza o desenvolvimento; no contexto DevOps/DataOps, ele se integra à automação e ao aprendizado contínuo.

SDLC (visão prática)

IdeiaPor que?

Definir problema, stakeholders e objetivo de negócio.

PlanejamentoComo?

Requisitos, roadmap e riscos. Sem isso, “caos a mais”.

DesenvolvimentoConstruir

Codificar com padrões (style guides) e revisão de código.

TesteValidar

Testes cedo e frequentes (Shift Left) para reduzir custo.

EntregaRelease

Empacotar e preparar a versão com rastreabilidade.

OperaçãoRun

Disponibilidade, observabilidade, resposta a incidentes.

ManutençãoMelhorar

Correções, melhorias e aprendizado para o próximo ciclo.

Ciclo de vida DevOps (8 etapas)

Um fluxo iterativo: planejamento → codificação → build → teste → lançamento → implementação → operar → monitorar.

Etapa Foco
Planejar Backlog e prioridades orientadas por feedback.
Codificar Pequenas mudanças, revisão por pares e padrões.
Build Integrar e empacotar; preparar artefatos.
Testar Testes automatizados e contínuos; linting.
Lançar Verificações finais e conformidade.
Implementar Deploy em produção com segurança e repetibilidade.
Operar Disponibilidade, performance e confiabilidade.
Monitorar Métricas, logs e feedback para o próximo ciclo.

Shift Left: qualidade começa cedo

Em vez de esperar as fases finais, o Shift Left “puxa” testes e validações para o início. Isso reduz retrabalho, surpresa e custo.

O que muda na prática

Antes: testar só no final → bugs caros e correções apressadas.

Depois: testar cedo e com frequência → feedback rápido e entregas seguras.

A CI é uma materialização prática do Shift Left: cada commit dispara validações automáticas antes de chegar à produção.

Tipos de testes (Shift Left)

unidade integração API/contrato IU

O objetivo não é “testar mais por testar”, e sim reduzir risco antes do deploy. Em dados, isso vira validação de schema, nulos, consistência de colunas e regras de qualidade.

Runbook: resposta consistente a incidentes

Runbooks reduzem pânico e improviso: definem sintomas, passos de diagnóstico, critérios de sucesso e escalonamento.

Identificação do problema (exemplo)

  • Sintoma: HTTP 500 (Internal Server Error) na API de cadastro.
  • Impacto: novos clientes não cadastrados, operação comercial travada, SLA em risco.
  • Alerta: escalar se persistir após os passos do runbook.

Possíveis causas (priorizar investigação)

  • Banco de dados fora do ar (conexão falhou / serviço inativo).
  • Bug em código recente (deploy introduziu erro).
  • Serviço da API parado (processo encerrado).

Passo a passo (5 etapas)

  1. Verificar se a API está ativa (ex.: endpoint de status).
  2. Checar logs do sistema e procurar erros recentes.
  3. Verificar conexão com o banco de dados (testar acesso).
  4. Reiniciar o serviço da API (ex.: npm start em cenários Node).
  5. Testar o endpoint de cadastro (ex.: POST /api/clientes).

Critério de sucesso: API volta a responder com HTTP 200, cadastro funciona, logs sem erros críticos e banco conectado.

Escalonamento: acionar responsável técnico, registrar incidente com hora/causa/ações e comunicar stakeholders.

Para modelagem e documentação de runbooks, uma boa referência é o template da Atlassian.

Projeto prático: CI/CD + GitHub Pages

Esta entrega simula um fluxo real DevOps: validar alterações no site (CI) e publicar automaticamente (CD).

Estrutura do repositório

Organização mínima conforme solicitado na atividade prática.

Arquivo/Pasta Função
site/index.html Página única do projeto.
.github/workflows/ci.yml Valida que o arquivo existe e faz checagens leves de consistência.
.github/workflows/cd.yml Publica a pasta site/ no GitHub Pages.

Onde ver evidências

Logs do pipeline: guia Actions no GitHub, com cada execução do workflow.

Artefato de deploy: o CD faz upload do conteúdo de site/ para publicar no Pages.

Resultado final: o site fica acessível por uma URL do GitHub Pages, com histórico de deploys.

Dica: no GitHub Actions, workflows podem ser disparados por push, pull request ou manualmente (workflow_dispatch).
